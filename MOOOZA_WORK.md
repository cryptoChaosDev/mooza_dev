# MOOOZA MVP

> Все шрифты должны быть бесплатными, все иконки, цвета и прочее — также бесплатные.

---

## Что получает пользователь в MVP

- Профиль-визитка
- Специалист получает клиентов и заказы
- Заказчик получает доступ к специалистам, находит с кем работать
- Причастность к музыкальной тусовке

---

## Минимальный сценарий MVP (user flow)

1. Пользователь регистрируется
2. Заполняет профиль
3. Добавляет услуги
4. Добавляет портфолио
5. Пользователь ищет специалиста
6. Открывает услугу
7. Нажимает «Заказать»
8. Переходит в чат
9. Обсуждают детали
10. Оформляют сделку
11. Исполнитель сдаёт работу
12. Заказчик закрывает сделку
13. Ставят оценки и оставляют отзывы
14. Выстраивают связи
15. Размещают посты в Потоке
16. Добавляют друзей

## Приоритеты разработки MVP

⚠️ Нужно исправить баг: при попытках добавить в друзья, связи и избранное — ничего не происходит.

⚠️ **Типы полей ввода:** по всему документу необходимо проставить формат получения данных для каждого поля, где пользователь что-то вводит или выбирает. Варианты форматов:
- Свободный ввод (текст)
- Свободный ввод с ограничением по количеству символов
- Числовой ввод
- Выбор одного значения из списка (single select / dropdown)
- Выбор нескольких значений из списка (multiselect)
- Поиск с ограничением по каталогу (autocomplete с выпадающим списком)
- Поиск с мультиселектом по каталогу
- Выбор даты (date picker)
- Переключатель (toggle / radio button)
- Чекбокс
- Загрузка файла

Эта разметка нужна для корректного описания интерфейса и технического задания для фронтенда. Задача: пройти по всем разделам документа и проставить типы полей там, где они ещё не указаны.

## Типы пользователей

В MVP доступна регистрация только одного типа профиля — **Резидент**. Остальные типы появятся в следующих версиях.

---

### 1. Резидент (Деятель музыкальной индустрии)

Специалист, работающий в музыкальной индустрии: музыкант, продюсер, звукорежиссёр, дизайнер, фотограф и т.д.

**Онбординг:** стандартный полный — имя, профессия (обязательно минимум одна из каталога), локация.

**Профиль:** полный функционал — профессии с фильтрами, услуги, портфолио, кейсы, связи, сделки, отзывы, соцсети, контакты.

**В каталоге «Люди»:** отображается с профессией, средней оценкой (если есть), связями.

---

### Типы профилей — будущие версии

Следующие типы профилей появятся в обновлениях после MVP. При попытке выбора любого из них при регистрации пользователь видит форму с предложением оставить email для уведомления о запуске.

**Слушатель / фанат** — человек, который следит за артистами и музыкальной сценой, не являясь профессионалом индустрии.

**Представитель компании** — сотрудник лейбла, продюсерского центра, площадки, студии и т.д.

**Заказчик** — человек, который хочет воспользоваться услугами платформы, не являясь деятелем индустрии.

🔗 Подробные описания этих типов — в разделе «Будущие функции».

### Незарегистрированный / неавторизованный пользователь

Пользователь, который не зарегистрировался или не вошёл в аккаунт, видит следующее:

**Поток:**

| Тип публикации | Доступность |
|---|---|
| Свободный блог | Полностью |
| Вопрос / обсуждение | Полностью |
| Опрос | Полностью (голосовать нельзя) |
| Апдейт услуги | Полностью |
| Апдейт занятости | Полностью |
| Остальные типы | Появятся в следующих версиях |

Лайки, реакции, комментарии — недоступны. При попытке взаимодействия всплывает окно «Необходимо зарегистрироваться».

**Профили пользователей:**
- Основная информация профиля (ФИ, ник, локация, профессия, о себе, счётчики) — видна
- Блок «Артисты», «Услуги», «Портфолио», «Отзывы», «Связи» — видны
- Блок «Контакты и соцсети» — виден как раздел, но вместо содержимого отображается заглушка: «Контакты доступны только зарегистрированным пользователям с заполненным профилем. Зарегистрируйтесь, чтобы посмотреть»
- Кнопки «Написать», «Установить связь», «Добавить в друзья», «Добавить в избранное» — отображаются, но при нажатии всплывает окно «Необходимо зарегистрироваться»

**Каталог:**
- Каталог услуг, каталог артистов, каталог людей — доступны для просмотра
- При попытке совершить недоступное действие (написать, оформить сделку, оставить комментарий) — всплывает окно «Необходимо зарегистрироваться»

**Недоступно полностью:**
- Сообщения
- Отношения (друзья, связи, избранное)
- Создание публикаций в Потоке
- Оформление сделок
- Любые действия, требующие авторизации

## Регистрация

- Регистрация по email
- Регистрация через VK
- Подтверждение кодом

**Поля при регистрации по email:**
- Email — свободный ввод с валидацией формата email
- Пароль — свободный ввод. Требования: минимум 8 символов, минимум 1 буква, 1 цифра и 1 спецсимвол
- Код подтверждения — 4 цифры, числовой ввод с маской. Срок действия кода: 15 минут
- Под полем ввода кода — кнопки «Запросить код повторно» и «Написать в поддержку»

**Согласия при регистрации:**
Внизу формы регистрации текст: «Регистрируясь, вы подтверждаете, что вам исполнилось 16 лет и что вы принимаете Пользовательское соглашение и Политику конфиденциальности» (со ссылками на документы).
- Чекбокс «Я даю согласие на обработку персональных данных» — **обязательный**
- Чекбокс «Я согласен получать рекламные рассылки» — необязательный

**Возрастные ограничения:**
- Регистрация доступна с 16 лет
- Финансовые операции (оплата сделок, выплаты, привязка карты) — с 18 лет
- Плашка с возрастным ограничением размещается в подвале сайта

**Если email уже зарегистрирован:**
Сообщение: «Данный email уже зарегистрирован» + предложение войти.

**Отзыв согласия на обработку персональных данных:**
Пользователь пишет на отдельный email (например, legal@moooza.ru). Команда оперативно обрабатывает и удаляет данные. Порядок прописывается в Политике конфиденциальности.

**Приоритет способов регистрации:**
1. VK
2. email

## Авторизация и вход

- Вход через VK
- Вход по email
- Восстановление пароля
- Повторный вход без потери сессии

**Восстановление пароля:**
1. Пользователь указывает email регистрации
2. Если email не найден — сообщение: «Такой email не зарегистрирован»
3. На email приходит письмо с кодом (срок действия: 15 минут)
4. Под полем ввода кода — кнопки «Запросить код повторно» и «Написать в поддержку»
5. После ввода кода открывается окно ввода нового пароля
6. Требования к паролю: минимум 8 символов, минимум 1 буква, 1 цифра, 1 спецсимвол

## Стартовое заполнение профиля

### Шаг 0 — Выбор типа профиля

Самый первый шаг перед регистрацией. Пользователь выбирает тип профиля из четырёх вариантов:
- Резидент (Деятель музыкальной индустрии)
- Слушатель / фанат
- Представитель компании
- Заказчик

**Если выбран «Резидент»:**
Пользователь переходит к стандартному сценарию регистрации (Карточка 1 — имя, далее).

**Если выбран любой другой тип:**
Открывается окно с текстом:

> «На данный момент доступна регистрация только для Резидентов — деятелей музыкальной индустрии.
>
> Профиль [тип, который выбрал пользователь] появится совсем скоро. Оставь свою контактную почту — мы обязательно уведомим тебя, как только регистрация будет доступна.»

Элементы окна:
- Поле ввода электронной почты
- Чекбокс «Согласен на получение уведомлений на почту»
- Кнопка «Отправить»

После нажатия «Отправить» — email сохраняется, пользователь получает подтверждение что его контакт принят.

### Карточка 1 — Имя, Фамилия, Ник, Аватарка

- Фамилия и Имя обязательны к заполнению, помечены звёздочкой
- Ник к заполнению необязателен
- Если вход через VK — данные подтягиваются оттуда, но пользователь может их изменить
- Внизу одна кнопка «Далее», неактивна до заполнения всей обязательной информации
- Ник должен быть уникальным и проверяться на уникальность. Если такой ник уже есть в системе — он выделяется красным и появляется надпись «Никнейм занят, введите другой»

⚠️ Проверка уникальности ника не реализована.

### Карточка 2 — Профессия

- Текстовое поле ввода с ограничением каталогом профессий
- Подсказка в строке ввода: «Начните вводить и выберите из выпавшего списка»
- При вводе текста открывается выпадающий список подходящих значений
- Пользователь выбирает профессию — раскрывается список соответствующих фильтров
- Пользователь может заполнить фильтры или пропустить этот шаг (заполнить позже в профиле) — об этом должна быть информация на экране
- Пользователь может добавить несколько профессий уже на этом шаге. Добавление хотя бы одной профессии — **обязательно**, завершить регистрацию без профессии нельзя
- На экране подсказка: «Если у вас более 1 профессии, остальные сможете добавить в профиле. Можно добавить до 10 профессий»

**Кнопки внизу экрана:**
- «Сохранить» — введённая информация сохраняется в профиле, переход на следующую карточку
- «Пропустить» — переход на следующую карточку, сохраняется только название профессии без фильтров

#### Сценарий: профессия не найдена в каталоге

Если пользователь ввёл запрос и не нашёл своей профессии в выпадающем списке, появляется кнопка **«Добавить свою профессию»** с подсказкой «Напишите нам, мы добавим профессию».

**Пошаговый сценарий:**

1. Пользователь нажимает «Добавить свою профессию»
2. Открывается форма запроса в саппорт — пользователь описывает свою профессию
3. Нажимает «Отправить»
4. Пользователю приходит подтверждение: «Мы приняли ваш запрос. Ожидайте ответ от саппорта. Вы сможете добавить профессию позже, когда она появится в каталоге»
5. Пользователь продолжает регистрацию и заполнение профиля — без указания отсутствующей профессии. Если у него есть другие профессии из каталога, они указываются. Регистрация завершается в полном объёме

**Обработка запроса саппортом:**

Саппорт рассматривает запрос вручную и принимает одно из решений:

| Решение | Действие саппорта | Уведомление пользователю |
|---|---|---|
| Профессия существует и относится к индустрии | Добавляет профессию в каталог | Письмо на указанный контакт: «Ваша профессия добавлена — продолжите регистрацию» |
| Профессия не относится к музыкальной индустрии (MVP) | — | Письмо: «На данный момент регистрация доступна только для представителей музыкальной индустрии» |
| Профессия не относится к индустрии (пост-MVP) | — | Письмо с предложением выбрать другой тип профиля |
| Другие случаи | На усмотрение саппорта | Индивидуальный ответ |

После получения уведомления об одобрении пользователь заходит в профиль и самостоятельно добавляет профессию из каталога.

🔗 Связано с: «Режим редактирования профиля» → блок «Профессии», «Уведомления», «Типы пользователей».

## Профиль пользователя

- **Обложка** — кнопка «Сменить фон», открывается системный компонент выбора фото/файлов
- **Аватарка** — кнопка смены аватарки, открывается системный компонент выбора фото/файлов
- **ФИ**
- **Ник**
- **Локация**
- **Статусы** (верифицировано и пр.) — тап по значку открывает подсказку. ⚠️ Статус «Pro» и «Амбассадор» — добавляются в первом обновлении после MVP
- **Профессия** — текстом через запятую, 2–3 строки. Текст цветом/шрифтом даёт понять, что он кликабельный. При нажатии раскрываются детали (фильтры + пользовательские фильтры), в правом верхнем углу крестик для закрытия окна
- **Счётчики** — Связи и Сделки
- **Кнопка шеринга** — иконка: квадратик со стрелкой вверх. При нажатии открывается системный компонент «Поделиться». В HTML добавить OG-теги для превью ⚠️ Иконка не та
- **Кнопка «Редактировать»** — профиль переходит в режим редактирования
- **О себе** — если блок заполнен, отображается текст; если не заполнен — блок отсутствует

### Артисты

Слайдер с плитками, на экране 3,5 или 4,5 плитки (чтобы было понятно, что слайдер листается).

## Сценарий жалобы на пользователя

**Кнопка «Пожаловаться»** находится в профиле другого пользователя.

**Пошаговый сценарий:**
1. Пользователь нажимает «Пожаловаться»
2. Открывается окно с заголовком «Опишите детали жалобы»
3. Пользователь выбирает категорию из выпадающего списка (single select — одна категория):
   - Спам
   - Недостоверная информация в профиле
   - Мошенничество / обман
   - Оскорбления и угрозы
   - Выдача себя за другого человека
   - Контент для взрослых / неприемлемый контент
   - Нарушение авторских прав
   - Другое
4. Заполняет текстовое поле — многострочный свободный ввод, без ограничения по символам
5. Нажимает «Отправить»
6. В этом же окне появляется подтверждение: «Ваша жалоба принята и будет рассмотрена в течение 3 дней»

**Куда поступает жалоба:** на email саппорта и в административную панель платформы.

**Анонимность:** жалоба анонимна — пользователь, на которого пожаловались, не видит кто подал жалобу.

**Решения саппорта:**

| Решение | Описание |
|---|---|
| Жалоба отклонена | Нарушений не найдено |
| Вынесено предупреждение | Нарушение зафиксировано, аккаунт остаётся активным |
| Контент удалён | Удалена конкретная публикация или информация в профиле |
| Аккаунт передан на дополнительную проверку | Требуется больше времени |
| Аккаунт заблокирован временно | Временная блокировка с указанием срока |
| Аккаунт заблокирован навсегда | Постоянная блокировка |

**Уведомления:**
- Подавший жалобу получает уведомление с финальным решением и причиной
- На кого пожаловались — не уведомляется при отклонении или удалении контента
- При предупреждении получает: «На вас поступила жалоба по категории [категория]. Вам вынесено предупреждение»
- При блокировке — см. «Сценарий блокировки аккаунта»

## Сценарий блокировки аккаунта

При вынесении решения о блокировке аккаунт немедленно деактивируется, активная сессия завершается.

**Что видит пользователь при попытке войти:**
- При попытке входа: «Профиль заблокирован»
- Временная блокировка: «Ваш аккаунт заблокирован до [дата]. Причина: [категория нарушения]»
- Постоянная блокировка: «Ваш аккаунт заблокирован навсегда. Причина: [категория нарушения]»

**Последствия блокировки:** профиль заблокированного становится неактивным и некликабельным везде — в каталоге, Потоке, избранном, связях, сообщениях. Двусторонняя связь не разрывается, но взаимодействие недоступно. Кнопки блокировки у пользователей нет — только жалоба.

**Уведомление:** отправляется на email, указанный при регистрации.

**Как оспорить:** через раздел «Информация» в шапке — написать в Telegram или на email саппорта. Решение об ответе — на усмотрение команды.

## Каталог

### Раздел Услуги

#### Структура каталога услуг

Каталог услуг организован в три уровня:

**Уровень 1 — Главная страница каталога услуг**

Вверху — строка поиска с ограничением по каталогу услуг (autocomplete). Пользователь вводит название услуги, под строкой появляется предварительная выдача. При выборе из выдачи — переход сразу на страницу конкретной услуги с карточками исполнителей.

Ниже строки поиска — плитки групп услуг (разделы верхнего уровня). Примеры: «Создание музыки», «Дизайн», «Запись», «Постпродакшн». Плитки кликабельны.

Ещё ниже — карточки всех услуг платформы вперемешку, отсортированные по умолчанию по дате добавления (последние — вверху).

**Уровень 2 — Страница группы услуг**

После нажатия на плитку группы (например, «Дизайн») открывается список конкретных услуг внутри группы. Примеры: «Дизайн мерча», «Дизайн обложки», «Дизайн логотипа». Отображаются плитками. Плитки кликабельны.

**Уровень 3 — Страница конкретной услуги**

После выбора конкретной услуги (например, «Дизайн обложки») открывается страница со всеми карточками исполнителей, оказывающих эту услугу.

#### Карточка услуги в каталоге

Каждая карточка содержит:
- Аватарка исполнителя
- Имя исполнителя
- Средняя оценка — отображается только если исполнитель был оценён хотя бы один раз. Если оценок нет — поле не отображается
- Название услуги (свободный ввод исполнителя, например «Звукореж от бога»)
- Стоимость от/до
- Первые 2–3 позиции прайс-листа с ценами
- Кнопка **«Написать»** — переход в личный чат с исполнителем
- Кнопка **«Оформить сделку»** — запуск сценария оформления сделки

Карточка кликабельна — переход на страницу услуги.

> ⚠️ **Репутация** — отдельный показатель, отличный от оценки. В MVP не реализуется, добавляется в следующих версиях. Оценка и репутация — два разных значения.

- **Стоимость** — диапазон от/до, числовой ввод

Можно выбрать несколько значений, каждое фиксируется тегом.

- **Локация** — поиск с ограничением по каталогу городов/стран + чекбоксы, мультиселект
- **Профессия** — поиск с ограничением по каталогу профессий + чекбоксы, мультиселект
- **Статус занятости** — фильтрация по статусу и дополнительным параметрам (см. «Статусы занятости»)

⚠️ Фильтры в MVP базовые, необходимо расширить в следующих версиях.

---

**Список участников:**

Над списком подпись «Все участники» и иконка сортировки.

⚠️ Счётчик количества участников рядом с подписью «Все участники» — убрать. Пока пользователей мало, цифра создаёт ощущение пустой платформы и психологически отпугивает. Счётчик включить после достижения 1000+ активных пользователей — на этом уровне он работает как социальное доказательство и играет в плюс.

**Сортировка** — при нажатии на иконку сортировки доступны варианты:
- **По релевантности** — по соответствию поисковому запросу и фильтрам (по умолчанию)
- **По оценке** — сначала пользователи с высокой средней оценкой
- **По количеству связей** — сначала наиболее «сетевые» пользователи
- **По алфавиту** — от А до Я / от Я до А

---

**Карточка пользователя в списке:**

Свёрнутый вид (всегда виден):
- Аватарка
- Фамилия и Имя
- Статусы рядом с именем (верифицирован и пр.). Статус «Pro» добавляется в первом обновлении
- Локация
- Профессии
- Количество связей
- Иконка-стрелка вниз по правому краю — показывает, что карточку можно раскрыть

---

**Раскрытое превью** (при нажатии на стрелку):

Занимает не более половины экрана телефона. Содержит:
- О себе — короткий текст из профиля
- Профессии — полный список
- Средняя оценка + количество отзывов (отображается только если есть оценки)
- Количество связей
- Кнопка «Перейти в профиль»

Раскрытое превью — это короткий просмотр ключевой информации без перехода на страницу профиля. Позволяет быстро оценить человека и принять решение о дальнейшем взаимодействии.

## Статусы занятости

Статус занятости отображается в профиле пользователя и используется как фильтр в каталоге «Люди». Доступен только для типа профиля «Деятель музыкальной индустрии».

> Статусы занятости и публикация «Апдейт занятости» в Потоке — доступны только Резидентам.

### Три статуса — принцип светофора

Выбор одного из трёх вариантов (radio button):

| Статус | Цвет | Описание |
|---|---|---|
| 🔴 **Закрыт** | Красный | Не берёт заказы, не рассматривает предложения |
| 🟡 **Рассматриваю** | Жёлтый | Смотрит предложения, но не в активном поиске |
| 🟢 **Открыт** | Зелёный | Готов к работе, ищет заказы |

Статус отображается в профиле пользователя в виде цветного кружка/плашки рядом с именем или профессией.

---

### Фильтры для статусов «Рассматриваю» и «Открыт»

При установке статуса «Рассматриваю» или «Открыт» пользователю открывается форма с дополнительными параметрами. Все три группы — **мультиселект** (можно выбрать несколько значений):

**География** (мультиселект):
- В своём городе
- В своём регионе
- По всей стране
- Международная занятость

**Тип занятости** (мультиселект):
- Постоянная
- Частичная (совмещение)
- Проектная
- Стажёр
- Волонтёр

**Тип оплаты** (мультиселект):
- Бесплатно / за респект
- Бартер
- Процент
- Ставка

Для статуса «Закрыт» дополнительных фильтров нет.

### Сетап

Сетап — перечень оборудования, которым располагает пользователь. Отображается в профиле и в статусе занятости. Набор позиций зависит от профессии пользователя. Тип поля: **мультиселект** (чекбоксы — пользователь отмечает то, что у него есть).

**Гитарист:**
- Гитара 6 струн
- Гитара 7 струн
- Гитара 8 струн
- Педалборд
- Домашняя студия
- Комбик
- Радиосистема
- Беспроводной ушной мониторинг
- Процессор эффектов

**Вокалист:**
- Микрофон концертный
- Микрофон студийный
- Домашняя студия
- Беспроводной ушной мониторинг
- Процессор эффектов

**Барабанщик:**
- Кардан
- Педаль
- Малый барабан
- Железо
- Стойки
- Акустическая ударная установка
- Электронная ударная установка
- Набор микрофонов
- Домашняя студия

**Все остальные инструменталисты:**
- Инструмент
- Домашняя студия
- Беспроводной ушной мониторинг
- Радиосистема

**Мультиинструменталист:** набор позиций формируется динамически — в зависимости от того, какие инструменты указаны в профиле. Для каждого инструмента подтягивается соответствующий стек (гитара → стек гитариста, барабаны → стек барабанщика, остальные → общий стек).

⚠️ Сетап нужен не всем профессиям. Список профессий, для которых сетап актуален, и полный перечень позиций для каждой профессии требуют отдельной проработки в привязке к каталогу профессий.

### Синхронизация статуса занятости

Статус занятости синхронизируется между профилем и Потоком:

**Публикация апдейта занятости в Потоке** → статус в профиле обновляется автоматически.

**Изменение статуса в профиле** → всплывает окно «Публиковать в Потоке?»:
- «Нет» — статус меняется только в профиле
- «Да» — открывается форма апдейта занятости с предзаполненным новым статусом. Пользователь добавляет текст (что именно ищет, что ему интересно) и публикует. Пост уходит в Поток, статус в профиле обновляется

## Поток

Поток — главная лента платформы. Все публикации пользователей отображаются в едином потоке. Каждая публикация имеет тип, который определяет её структуру и назначение.

---

### Интерфейс ленты

**Сортировка по умолчанию:** по дате и времени добавления — самые новые посты вверху. Дополнительные варианты сортировки доступны через иконку сортировки.

**Лента для незарегистрированных пользователей** выглядит так же, как для зарегистрированных — визуальных отличий нет. Отличается только доступность действий (лайки, комментарии, репост недоступны).

---

### Карточка публикации в ленте

Каждый пост в ленте отображается в следующем порядке:

1. **Шапка поста:**
   - Аватарка автора
   - Фамилия и Имя (или название канала/артиста)
   - Бейджи рядом с именем (верифицирован и пр.). Статус «Pro» добавляется в первом обновлении

2. **Тип публикации** — плашка с типом (Вопрос, Опрос, Апдейт услуги и т.д.). Для свободного блога плашка не отображается

3. **Содержание публикации** — текст, изображения, элементы в зависимости от типа

4. **Кнопки действий под постом:**
   - Лайк / анлайк
   - Комментировать — комментарии раскрываются прямо под постом в ленте
   - Репост ⚠️ Логика репоста требует обсуждения с командой
   - Поделиться
   - Пожаловаться (см. «Сценарий жалобы на публикацию»)
   - ⭐ Звёздочка в правом верхнем углу поста — добавить в сохранённые

---

### Навигация и фильтрация ленты

Сразу под названием раздела — два ряда чипсов. Каждый ряд можно скроллить горизонтально.

**Ряд 1 — Тип публикации:**
- Все (по умолчанию)
- Свободный блог
- Вопрос / обсуждение
- Опрос
- Апдейт услуги
- Апдейт занятости
- *(остальные типы добавляются по мере выхода)*

**Ряд 2 — Автор публикации:**
- Все (по умолчанию)
- Резидент *(публикации от физлиц)*
- Канал
- Артист
- Компания
- Заказчик
- Мои *(только публикации текущего пользователя, доступно только авторизованным)*

Под чипсами — иконки **Сортировка** и **Фильтры**.

**Сортировка:**
- По новизне (по умолчанию) — последние публикации вверху
- По популярности — больше всего реакций и комментариев за последние 24–48 часов
- По релевантности — на основе профессий, жанров и активности пользователя (только для авторизованных)

**Фильтры** — динамические, меняются в зависимости от выбранного типа публикации и типа автора.

*Базовые фильтры (всегда доступны):*
- Жанр
- Город / регион
- Профессия автора

*Фильтры по типу публикации:*

| Тип публикации | Дополнительные фильтры |
|---|---|
| Вопрос / обсуждение | Категория вопроса |
| Опрос | Статус: активные / завершённые |
| Апдейт услуги | Тип апдейта, раздел каталога |
| Апдейт занятости | Статус занятости, профессия |

*Фильтры по типу автора:*

| Тип автора | Дополнительные фильтры |
|---|---|
| Резидент | Профессия, город, жанр |
| Канал | Тематика канала |
| Артист | Жанр, город |

⚠️ Интерфейс ленты и алгоритм ранжирования требуют отдельного описания.

**Поток для нового пользователя:** при первом входе лента наполнена постами команды сервиса Moooza — чтобы пользователь не видел пустой экран.

### Общая логика всех публикаций

Любая публикация в Потоке состоит из:
- Автора публикации
- Типа публикации
- Контента
- Метаданных
- Взаимодействий

---

### Автор публикации

При создании поста пользователь выбирает, от чьего имени публиковать:

1. **Профиль физлица** — пост от самого человека. Пример: Антон Сёмин
2. **Канал физлица** — тематический канал пользователя. Примеры: «Дизайн со слоном», «Сведение без боли», «Дневник турового музыканта»
3. **Канал артиста** — если пользователь является администратором. Примеры: IDEЯ FIX, сольный артист, музыкальный проект
4. **Канал юрлица** — не в MVP, добавляется позже

---

### Общие элементы любой публикации

**Обязательные:**
- Тип публикации
- Автор
- Текст публикации (многострочный свободный ввод)

**Опциональные:**
- Фото / изображения — загрузка файла, до 10 изображений
- Теги — свободный ввод с символом #, пользователь сам пишет теги
- Жанры — поиск с ограничением по каталогу музыкальных жанров, мультиселект (autocomplete + выпадающий список)
- Город — поиск с ограничением по каталогу, autocomplete
- Ссылки — свободный ввод текста
- Упоминания пользователей, каналов, артистов — через @ с поиском по базе платформы

**Действия под любой публикацией:**
- Реакции
- Комментарии и ответы на комментарии
- Репост
- ⭐ Звёздочка в правом верхнем углу — добавить в сохранённые
- Поделиться
- Пожаловаться
- Перейти в профиль / канал автора

### Типы публикаций — MVP

---

#### 1. Свободный блог

**Назначение:** свободный пост без жёсткой структуры. Для мыслей, backstage, фото со студии, историй, анонсов, контента сцены, личного и профессионального контента.

**Поля формы:**

| Поле | Обязательное | Описание |
|---|---|---|
| Текст публикации | Да | Главное поле, свободный ввод |
| Фото / изображения | Нет | До 10 изображений |
| Теги | Нет | Например: #студия, #репетиция, #tour |
| Жанры | Нет | Например: alternative rock, metalcore |
| Город | Нет | — |
| Ссылки | Нет | Внешние ссылки |
| Упоминания | Нет | Пользователей, артистов, каналов |

**Отображение поста:**
- Аватар и имя автора, тип автора
- Текст
- Изображения
- Реакции и комментарии

---

#### 2. Вопрос / обсуждение

**Назначение:** пост для получения советов, рекомендаций, мнений, запуска обсуждений.

**Поля формы:**

| Поле | Обязательное | Описание |
|---|---|---|
| Заголовок вопроса | Да | Например: «Где сводить live-барабаны в Москве?» |
| Текст вопроса | Да | Описание ситуации |
| Категория вопроса | Нет | ⚠️ Single select или мультиселект — требует уточнения |
| Фото / изображения | Нет | — |
| Жанры | Нет | — |
| Город | Нет | — |
| Теги | Нет | — |

**Особенности:**
- Акцент на комментариях
- Можно сортировать по активности
- В будущем: кнопка «Решено»

**Отображение:** плашка «Вопрос»

---

#### 3. Опрос

**Назначение:** голосование сообщества.

**Поля формы:**

| Поле | Обязательное | Описание |
|---|---|---|
| Вопрос опроса | Да | Например: «Какую обложку выбрать?» |
| Варианты ответов | Да | ⚠️ Тип поля, ограничения по количеству вариантов и символам — требуют уточнения |
| Длительность опроса | Да | ⚠️ Фиксированные варианты или ввод вручную — требует уточнения |
| Фото / изображения | Нет | — |
| Текст публикации | Нет | Дополнительный контекст |
| Теги | Нет | — |

**Особенности:**
- Голосование с отображением процентов и количества голосов
- После окончания срока голосование закрывается

**Отображение:** плашка «Опрос»

---

#### 4. Апдейт услуги

**Назначение:** публикация, связанная с существующей услугой пользователя. Показывает активность вокруг услуги — обновления, новые кейсы, свободные слоты, изменения доступности. Услуга создаётся не в Потоке и должна существовать заранее.

**Поля формы:**

| Поле | Обязательное | Описание |
|---|---|---|
| Выбор услуги | Да | Выбор из опубликованных услуг пользователя |
| Тип апдейта | Нет | Свободные места / Новый кейс / Обновление портфолио / Изменение доступности / Временное предложение / Другое |
| Текст публикации | Да | Например: «Освободилось 2 места на июнь» |
| Фото / изображения | Нет | — |
| Прикрепить кейс | Нет | Из портфолио или кейсов услуги |

> Тип апдейта — опциональный быстрый тег. Помогает системе фильтровать и ранжировать апдейты в ленте, а читателю — сразу понять суть ещё до прочтения текста.

**Ограничения:** не чаще 1 раза в 24 часа на одну услугу. Услуга должна быть опубликована. Одинаковые тексты система не пропускает автоматически. Пользователи могут пожаловаться на спам.

**Отображение:** плашка «Апдейт услуги» + название привязанной услуги + кнопки «Детали услуги», «Написать», «Оформить сделку»

---

#### 5. Апдейт занятости

**Назначение:** публичное изменение профессиональной доступности пользователя. Это не вакансия, не услуга и не поиск сотрудника — это профессиональный статус доступности.

> ⚠️ **Ограничение по автору:** апдейт занятости можно опубликовать только от имени профиля физлица. При выборе канала физлица, канала артиста или канала юрлица этот тип публикации недоступен.

**Поля формы:**

| Поле | Обязательное | Описание |
|---|---|---|
| Новый статус | Да | Закрыт / Рассматриваю / Открыт — radio button, выбор одного из трёх |
| Текст публикации | Да | Например: «Освободился на июль. Ищу проекты по live-съёмке и backstage-фото» |
| Фото / изображения | Нет | — |
| Жанры | Нет | — |
| Профессии | Нет | — |
| Город / онлайн | Нет | — |

**Автоматически подтягивается:** профессии пользователя, новый статус, ссылка на профиль.

**Отображение:** плашка «Апдейт занятости»

🔗 Связано с: «Статусы занятости», «Синхронизация статуса занятости».

---

### Типы публикаций — первое обновление после MVP

##### 6. Событие
Концерты, фестивали, встречи, лекции, кастинги, мастер-классы. ⚠️ Структура требует проработки.

##### 7. Вакансия / поиск человека
Поиск специалистов и участников. Примеры: ищем барабанщика, нужен дизайнер, ищем FOH. ⚠️ Структура требует проработки.

##### 8. Коллаборация
Поиск совместного творчества. Коллаборация ≠ проект, это social matchmaking. Примеры: ищу соавтора, собираю группу, ищу людей в тур. ⚠️ Структура требует проработки.

##### 9. Кейс
Публикация о выполненной работе: разбор проекта, результат, процесс, до/после, backstage. ⚠️ Структура требует проработки. 🔗 Связано с: «Кейсы на странице услуги».

---

### Типы публикаций — бэклог

##### 10. Продажа / аренда
Продажа инструментов, оборудования, техники, мерча. Аренда репетиционных баз и помещений. ⚠️ Структура требует проработки.

##### 11. Релиз

## Уведомления

### Общее описание

Уведомления доступны всем зарегистрированным пользователям, включая администраторов. Администратор получает уведомления в той же системе, что и обычные пользователи.

**Канал доставки:**
- In-app — внутри платформы, в разделе «Уведомления» в шапке
- Push-уведомления на телефон

**Хранение:** уведомления хранятся бессрочно.

**Настройки:** на текущем этапе управление уведомлениями недоступно. Настройки типов и каналов уведомлений будут добавлены в следующих версиях.

---

### Интерфейс

**Иконка в шапке:**
- Отображает счётчик непрочитанных уведомлений ⚠️ Формат счётчика (цифра или точка) уточняется
- При нажатии раскрывается список уведомлений

**Список уведомлений:**
- Непрочитанные — выделены жирным шрифтом
- Прочитанные — обычный шрифт
- Сортировка: по дате и времени поступления, вверху — последние
- Все уведомления сохраняются, ничего не удаляется автоматически
- Кнопка «Прочитать всё» — отмечает все уведомления как прочитанные

## Верификация артиста

Все артисты в каталоге верифицированы — неверифицированные артисты в каталоге не отображаются. Верификация подтверждает, что пользователь действительно является представителем артиста и имеет доступ к его официальным соцсетям.

---

### Сценарий создания и верификации артиста

1. Пользователь нажимает «Создать профиль Артиста»
2. Открывается форма. Вверху — **инструкция**:
   > «Для размещения артиста в каталоге необходимо пройти верификацию. Вы получите уникальный код, который нужно разместить в описании профиля артиста в соцсетях и прислать нам ссылку на этот профиль. Саппорт проверит код и опубликует карточку.»

3. Пользователь выбирает свою роль — **single select**:
   - Музыкант
   - Менеджер
   - Директор
   - Представитель группы
   - Лейбл

4. Пользователь заполняет карточку артиста ⚠️ Состав полей карточки требует описания в разделе «Профиль Артиста»

5. Пользователь нажимает кнопку **«Получить код»**

6. Система генерирует уникальный верификационный код и отображает его пользователю

7. Пользователь размещает код в **описании профиля** артиста в соцсетях (ВКонтакте, Instagram и т.д.)

8. Пользователь вставляет ссылку на профиль артиста в соцсетях в поле **«Ссылка для верификации»** — свободный ввод URL

9. Нажимает **«Отправить на верификацию»** — запрос уходит в саппорт

10. Пользователь получает подтверждение: «Ваш запрос отправлен. Мы проверим и уведомим вас о результате»

---

### Обработка верификации саппортом

Саппорт получает запрос, переходит по ссылке и визуально проверяет наличие кода в описании профиля артиста в соцсетях.

**Если код найден:**
- Карточка артиста публикуется в каталоге
- Пользователь получает уведомление: «Артист [Название] успешно верифицирован и добавлен в каталог»
- Пользователь становится администратором профиля артиста

**Если код не найден или ссылка недействительна:**
- Карточка не публикуется
- Пользователь получает уведомление с причиной и возможностью повторить попытку

---

### Логика отображения

- Артист появляется в каталоге только после успешной верификации
- До верификации карточка существует в системе, но не видна другим пользователям
- У пользователя-администратора карточка отображается в профиле со статусом «На верификации»

🔗 Связано с: «Профиль Артиста», «Сценарий добавления артиста», «Уведомления».

### Сценарий добавления релиза

1. Администратор артиста нажимает кнопку «Добавить» в разделе «Релизы»
2. Открывается форма добавления релиза
3. Пользователь выбирает платформу — **single select**: ВКонтакте, Spotify, Яндекс Музыка, Apple Music

5. Система автоматически подтягивает из стриминга: название релиза, обложку, дату релиза. **Требование к разработчику:** определить способ реализации автоподтягивания данных

7. Нажимает «Сохранить»

**После нажатия «Сохранить»:**
- Релиз появляется в разделе «Релизы» профиля артиста

- Если участник нажимает «Подтвердить» → администратор артиста получает уведомление о подтверждении. Участник отображается в разделе участников релиза
- Если участник нажимает «Отклонить» → администратор получает уведомление об отказе. Участник не отображается в релизе
- ⚠️ Что отображается у участника в его профиле после подтверждения участия в релизе — обязательно к определению и заполнению

## Сделки

### Типы услуг

Все услуги в каталоге делятся на два типа. Тип указывается на бэкенде в каталоге услуг при заведении услуги командой Moooza. На карточке услуги и на странице услуги отображается тег типа — **«Процессная»** или **«Событийная»** — чтобы заказчик сразу понимал логику взаимодействия.

**Тип А — Процессная услуга**
Исполнитель работает в течение срока сделки, результат сдаётся по завершении.
Примеры: дизайн, сведение, запись, монтаж, написание текстов.

**Тип Б — Событийная услуга**
Исполнитель оказывает услугу в конкретную дату. До этой даты он резервирует время и отказывается от других заказов.
Примеры: звукорежиссура концерта, фотограф на съёмке, ведущий мероприятия, билетёр.

Логика статусов, оплаты, приёмки и отмены — разная для каждого типа.

---

### Тип А — Процессная сделка

#### Статусы сделки типа А

| Статус | Кто переводит | Условие перехода | Следующий статус |
|---|---|---|---|
| **На согласовании** | Система | Сделка оформлена и отправлена | → «Ожидает оплаты» (User2 принял) / → «Отменена» (кто-либо отменил или User2 отклонил) |
| **Ожидает оплаты** | Система | User2 подтвердил сделку | → «В работе» (заказчик оплатил) / → «Отменена» (кто-либо отменил) |
| **В работе** | Система | Заказчик произвёл оплату | → «На проверке» (исполнитель нажал «Сдать работу») / → «Отменена» (кто-либо отменил или истёк срок сдачи) |
| **На проверке** | Система | Исполнитель нажал «Сдать работу» | → «Завершена» (заказчик принял или истёк срок приёмки) / → «На доработке» (заказчик отправил на доработку) / → «Отменена» (кто-либо отменил) |
| **На доработке** | Система | Заказчик нажал «На доработку» | → «На проверке» (исполнитель повторно нажал «Сдать работу») / → «Отменена» (кто-либо отменил) |
| **Завершена** | Система | Заказчик нажал «Принять» или истёк срок приёмки без ответа | — (финальный статус) |
| **Отменена** | Система | Отмена любым участником, отклонение, истечение срока сдачи | — (финальный статус) |

**Автоматические переходы:**
- «В работе» → «Отменена»: если истёк срок сдачи работы, а исполнитель не нажал «Сдать работу». Заказчику возвращаются деньги за вычетом комиссий платёжного оператора
- «На проверке» → «Завершена»: если истёк срок приёмки работы, а заказчик не совершил никаких действий. Деньги выплачиваются исполнителю, Moooza получает комиссию

**Уведомления при отмене:** оба участника получают уведомление об отмене вне зависимости от того, кто её инициировал.

---

#### Карточка сделки типа А — поля и обязательность

| Поле | Тип ввода | Обязательное | Примечание |
|---|---|---|---|
| Название сделки | Свободный ввод | Да | Автозаполняется из карточки услуги, можно изменить |
| Услуга (раздел каталога) | Привязка к каталогу | Да | Автозаполняется из карточки услуги |

#### Статусы сделки типа Б

| Статус | Кто переводит | Условие перехода | Следующий статус |
|---|---|---|---|
| **На согласовании** | Система | Сделка оформлена и отправлена | → «Ожидает оплаты» (User2 принял) / → «Отменена» (кто-либо отменил или User2 отклонил) |

**Автоматический переход:**
- «Ожидает подтверждения» → «Завершена»: если в течение 3 дней после даты события заказчик не совершил никаких действий. Деньги выплачиваются исполнителю, Moooza получает комиссию.

#### Логика оплаты и депозита типа Б

При оформлении сделки типа Б заказчик оплачивает **полную стоимость** — деньги холдируются на виртуальном счёте платёжного сервиса до завершения или отмены.

**При завершении сделки:**
- Полная сумма за вычетом комиссии Moooza выплачивается исполнителю

**При отмене сделки:**
- Депозит + комиссия Moooza с суммы депозита → исполнителю и платформе
- Остаток (полная сумма − депозит − комиссия платёжного оператора) → возвращается заказчику

- После наступления даты события отмена недоступна
- При отмене: депозит остаётся у исполнителя, остаток возвращается заказчику

### Логика расчёта суммы и комиссии

**Комиссия платформы:** 10% от цены для клиента.

**Гонорар исполнителя:** 90% от суммы заказа. Комиссию платёжного оператора при выплате платит платформа из своих 10% — исполнитель всегда получает ровно 90%.

**При создании услуги:**
В полях стоимости и прайс-листа отображается подсказка: «Не забудьте учесть комиссию сервиса».

**При оформлении сделки:**
Исполнитель вводит сумму заказа. Больше ничего не отображается и не рассчитывается на этом этапе.

**На странице сделки (после оформления):**
- Тип А: Стоимость заказа, Гонорар исполнителя
- Тип Б: Стоимость заказа, Гонорар исполнителя, Сумма невозвратного депозита

**Валюта:** все расчёты в рублях (RUB).

### Платёжный сервис и безопасная сделка

**Механизм безопасной сделки (холдирование):**
1. Заказчик производит оплату — деньги поступают в платёжный сервис
2. Платёжный сервис создаёт виртуальный счёт, на котором деньги удерживаются (холдируются) до завершения или отмены сделки
3. При переходе сделки в статус «Завершена» — система отправляет сигнал в платёжный сервис, деньги размораживаются и распределяются:
   - Сумма на руки → на реквизиты исполнителя
   - Комиссия → на счёт Moooza
4. При переходе в статус «Отменена» (если оплата была произведена) — деньги возвращаются заказчику за вычетом комиссий платёжного оператора

### Реквизиты исполнителя — токенизация карты через платёжный сервис

> ⚠️ **Пост-MVP.** Весь раздел ниже (токенизация, выплаты, привязка карты) реализуется после запуска MVP при подключении платёжного сервиса.

Данные банковских карт исполнителей хранятся исключительно в инфраструктуре платёжного сервиса и никогда не передаются на серверы Moooza. Это исключает риски утечки конфиденциальных финансовых данных.

**Процесс привязки карты исполнителя (пошагово):**

1. Исполнитель нажимает кнопку «Указать данные для получения выплат» (в профиле пользователя)
2. Фронтенд загружает защищённый JS-виджет от платёжного сервиса — на экране появляется форма ввода номера карты и срока действия
3. Исполнитель вводит данные карты — виджет отправляет их напрямую в платёжный сервис, минуя серверы Moooza
4. Платёжный сервис проверяет карту, сохраняет её в своём защищённом хранилище и возвращает безопасный идентификатор — `payment_method_id` (токен)
5. Фронтенд передаёт полученный `payment_method_id` на бэкенд Moooza — токен сохраняется в базе данных и привязывается к профилю исполнителя

**Схема:**
```
[Исполнитель] → [Виджет платёжного сервиса на сайте] → [Серверы платёжного сервиса]
                                                                |
[БД Moooza] ← [payment_method_id (токен)] ←-------------------+
```

**Процесс выплаты при завершении сделки:**

При переходе сделки в статус «Завершена» сервер Moooza отправляет в API платёжного сервиса запрос на создание выплаты (`POST /payouts`), передавая:
- Сумму выплаты (сумма на руки исполнителю)
- `deal_id` — идентификатор сделки, из которой берутся холдированные средства
- `payment_method_id` — токен карты исполнителя

Платёжный сервис находит по токену реальный номер карты в своей базе и переводит деньги исполнителю.

**Валюта:** все расчёты ведутся в рублях (RUB).

**Фискальные чеки (54-ФЗ):**
При оплате заказчик вводит свой email — на него платёжный сервис отправляет фискальный чек. Email вводится вручную в форме оплаты при каждой транзакции.

**Возврат при отмене сделки:**
Если сделка переходит в статус «Отменена» после того, как оплата была произведена, заказчику возвращаются деньги за вычетом комиссии платёжного оператора. Сумма удержания зависит от суммы изначальной операции (нефиксированный процент платёжного сервиса).

**Важно для будущего масштабирования (юридические лица):**

Описанный механизм работает только для физических лиц (выплаты на карты). При добавлении юрлиц (ООО, ИП) процесс изменится: вместо виджета используется текстовая форма для ввода банковских реквизитов (ИНН, БИК, расчётный счёт). Реквизиты юрлиц не подпадают под стандарт PCI DSS, поэтому их можно хранить в БД Moooza. При выплате вместо токена карты в API платёжного сервиса передаётся массив банковских реквизитов.

### Связи сущностей

- **Сделка ↔ Услуга:** сделка привязана к конкретной карточке услуги в момент оформления. Данные из услуги копируются в сделку. Если исполнитель впоследствии переносит услугу в архив — сделка сохраняется со своими данными без изменений. Удаление опубликованных услуг не предусмотрено — только перевод в архив
- **Сделка ↔ Деловой чат:** деловой чат является отдельной сущностью, привязанной к сделке. Создаётся в момент перехода сделки в статус «В работе». Название чата = название сделки. При завершении или отмене сделки (если чат уже был создан, т.е. статус дошёл до «В работе») чат автоматически переходит в архив раздела «Деловые». При отмене на статусах «На согласовании» или «Ожидает оплаты» чат ещё не создан — архивировать нечего
- **Сделка ↔ Пользователи:** каждая сделка привязана к двум пользователям с зафиксированными ролями — Исполнитель и Заказчик
- **Сделка ↔ Связь между пользователями:** при завершении сделки между участниками автоматически устанавливается профессиональная связь (см. «Автоматическое установление связи»)

При оформлении сделки данные из карточки услуги (название, раздел каталога, фильтры) копируются непосредственно в запись сделки. Это гарантирует, что последующие изменения или архивация услуги не влияют на отображение данных в сделке.

**Деловой чат:**
Создаётся как отдельная запись в таблице `Chat` с типом `business` и привязкой `deal_id`. При переходе сделки в `completed` или `cancelled` — чат переводится в архив (флаг `is_archived = true`).

### Прайс-лист

Прайс-лист позволяет исполнителю детализировать стоимость услуги по конкретным позициям.

**Соотношение с полем «Стоимость»:**
- Поле «Стоимость от/до» — общий диапазон цены на услугу в целом.
- Прайс-лист — детализация: отдельные позиции с ценами внутри услуги.

*Пример: услуга «Дизайн мерча», стоимость от 500 до 5000 руб. В прайс-листе: дизайн принта футболки — 500 руб., составление техкарты для производства — 2000 руб., консультация — 5000 руб.*

**Структура одной позиции:**
- Текстовое поле — название позиции, ограничение 100 символов

**Добавление позиций:** под последней позицией — кнопка «+». По ней добавляется новая пустая строка. Количество позиций не ограничено.

Рядом с блоком прайс-листа подсказка: «Не забудьте учесть комиссию сервиса».

**Редактирование:** пользователь может добавлять новые позиции, редактировать текст и цену существующих, удалять позиции.

### Автоподтягивание фильтров из профиля

Каждая услуга в каталоге привязана к определённой профессии. При создании услуги система проверяет профиль пользователя:

- Если профессия, привязанная к выбранной услуге, указана в профиле **и** по ней заполнены фильтры-атрибуты — пересекающиеся фильтры в форме создания услуги заполняются автоматически. Пользователь может изменить подтянутые значения вручную.
- Если профессия не указана в профиле **или** указана, но фильтры-атрибуты не заполнены — соответствующие поля фильтров остаются пустыми и заполняются вручную.

🔗 Связано с: «Каталог профессий», «Каталог услуг», «Профиль пользователя» → «Профессии».

## Сценарии

### Сценарий добавления услуги

При нажатии на кнопку «Добавить» открывается на весь экран форма со следующими элементами:

- Название услуги — свободный ввод, ограничение 50 символов ⚠️ Не реализовано
- Раздел каталога — поиск с ограничением по каталогу услуг; пользователь вводит название и выбирает из выпадающего списка
- Детали услуги — фильтры-атрибуты, определяются выбранным разделом каталога:
  - Фильтр **«Уровень»** — single select (выбор одного значения): Новичок / Любитель / Профессионал / Эксперт
  - Все остальные фильтры — мультиселект (можно выбрать несколько значений)
  - Частично автозаполняются из профиля пользователя (см. «Автоподтягивание фильтров»)
- Стоимость — два числовых поля ввода: «от» и «до». Если не заполнены — отображается «По договорённости». Подсказка: «Не забудьте учесть комиссию сервиса»
- Прайс-лист — детализация стоимости по позициям (см. «Прайс-лист»). Подсказка: «Не забудьте учесть комиссию сервиса»
- Срок исполнения — два числовых поля ввода: «от» и «до». Пользователь вводит цифру вручную. Единица измерения — дни. Подсказка: «Укажите срок в днях»
- Описание — многострочное текстовое поле, свободный ввод, без ограничения по символам

**Кнопки формы:**
- «Отменить» — карточка услуги удаляется
- «Убрать в черновики» — карточка сохраняется в разделе «Черновики». При случайном закрытии страницы карточка также сохраняется в черновиках ⚠️ Сохранение в черновик не реализовано
- «Опубликовать» — запускает сценарий публикации услуги

### Редактирование услуги и публикация апдейта в Поток

При сохранении изменений в карточке услуги (кнопка «Сохранить» в режиме редактирования) система предлагает опубликовать апдейт в Поток:

> **«Сообщить об изменениях в Потоке?»**
>
> **Кнопка «Нет»** — изменения сохраняются, пост в Потоке не создаётся
>
> **Кнопка «Сделать апдейт»** — открывается форма апдейта услуги (см. «Апдейт услуги»). Пользователь заполняет текст апдейта, нажимает «Опубликовать» — пост уходит в Поток

При этом действуют все ограничения апдейта: не чаще 1 раза в 24 часа на одну услугу, уникальность текста, услуга должна быть опубликована.

🔗 Связано с: «Апдейт услуги», «Поток», «Режим редактирования услуги».

### Сценарий публикации услуги

После нажатия «Опубликовать» карточка услуги отправляется на модерацию. После прохождения модерации показывается диалог выбора:

> **Кнопка «В Потоке»** — открывается форма публикации поста об услуге (см. «Публикация услуги в Потоке»). После заполнения и отправки: услуга появляется в профиле и каталоге, пост публикуется в Потоке. Счётчик услуг +1.

### Модерация

> Основа раздела — план реализации системы модерации v1.1, подготовленный с учётом юридических замечаний. Документ является продуктово-аналитическим и техническим планом. Финальные формулировки пользовательских документов, рекламных процедур и регламентов должен утвердить юрист.

#### Ключевой принцип

Публичный контент проверяется системно. Спорный, рекламный, авторский и пожалованный контент разбирается человеком. Личные сообщения не читаются тотально.

---

#### Объекты модерации

**Публичный контент:**

| Объект | Состав | Режим |
|---|---|---|
| Профиль | Имя, ник, аватар, описание, ссылки, контакты, фото/видео/аудио | Автофильтр + постмодерация по риску |
| Карточка услуги | Описание, цена, примеры работ, ссылки, обложки | Автофильтр + выборочная премодерация по риску |
| Пост в Потоке | Текст, фото, видео, аудио, ссылки, хэштеги | Автофильтр + постмодерация / премодерация по риску |
| Комментарий | Текст, ссылки, упоминания | Автофильтр + жалобы |
| Отзыв | Оценка, текст, факты сделки | Усиленный контроль — риск клеветы и конфликтов |
| Медиа | Фото, видео, аудио, обложки, портфолио | Фильтры контента + авторские права |
| Платное продвижение | Промопосты, баннеры, рекламные карточки | Обязательная премодерация + проверка маркировки |

**Личные сообщения** — не читаются вручную в обычном режиме. Реализуются автоматические меры защиты и механизм жалобы. Ручной просмотр только по основанию.

---

#### Роли модерации

| Роль | Ответственность |
|---|---|
| Модератор L1 | Типовые жалобы, спам, оскорбления, простые нарушения, скрытие/восстановление |
| Старший модератор L2 | Повторные нарушения, спорные отзывы, блокировки, апелляции, жалобы на L1 |
| Юрист | Сложные правовые кейсы, авторские споры, реклама, запросы госорганов |
| Служба поддержки | Коммуникация с пользователем, сбор документов |

---

#### Общий поток публикации

1. Пользователь отправляет контент
2. Система сохраняет черновик
3. Запускаются Rule Engine, Link Scanner, Media Scanner
4. Рассчитывается `risk_score` и набор `moderation_flags`
5. Принимается решение:
   - `publish` — публикация сразу
   - `publish_with_post_review` — публикация + постмодерация
   - `pending_review` — не публиковать до проверки
   - `hidden_pending_review` — скрыть до проверки
   - `rejected` — отказ в публикации
   - `requires_legal_assessment` — эскалация юристу
6. Пользователь получает уведомление если контент ограничен
7. Все события записываются в Audit Log

---

#### Статусы контента

| Статус | Значение |
|---|---|
| `draft` | Черновик, не опубликован |
| `published` | Опубликован и доступен |
| `published_post_review` | Опубликован, в очереди постмодерации |
| `pending_review` | Ожидает ручной проверки до публикации |
| `hidden_pending_review` | Временно скрыт до проверки |
| `rejected` | Отклонён до публикации |
| `removed` | Удалён по решению модерации |
| `restored` | Восстановлен после проверки или апелляции |
| `requires_legal_assessment` | Требуется оценка старшего модератора / юриста |
| `blocked_by_system` | Автоматически заблокирован (спам, боты, фишинг) |

---

#### Категории флагов

| Флаг | Описание |
|---|---|
| `suspected_extremism` | Признаки экстремистских материалов. Ограничить и эскалировать |
| `suspected_terrorism` | Террористическое содержание. Ограничить и эскалировать |
| `suspected_violence_or_threat` | Угрозы, призывы к насилию |
| `suspected_pornography` | Порнографический или явно сексуальный контент |
| `suspected_child_safety` | Риски связанные с несовершеннолетними. Максимальный приоритет |
| `suspected_illegal_drugs` | Запрещённые вещества, способы приобретения |
| `suspected_fraud` | Мошеннические схемы, фишинг, обман |
| `suspected_copyright` | Возможное нарушение авторских / смежных прав |
| `suspected_personal_data` | Чужие персональные данные или документы |

#### Risk score

Контент не блокируется только по количеству жалоб — используется риск-скоринг для защиты от злоупотреблений и конкурентных атак.

**Факторы:**
- Тип контента (реклама, отзывы, аудио — повышающий коэффициент)
- Категория жалобы (экстремизм, угрозы, детская безопасность — высокий приоритет)
- Репутация жалобщика (история ложных жалоб, возраст аккаунта)
- История автора (повторные нарушения, подозрительная активность)
- Автоматические флаги (стоп-слова, вредоносные ссылки, OCR)
- Контекст сделки (для отзывов — была ли сделка внутри платформы)

**Матрица решений:**

| Балл | Риск | Действие |
|---|---|---|
| 0–20 | Низкий | Опубликовать без ручной проверки |
| 21–45 | Средний | Опубликовать + отправить в постмодерацию |
| 46–70 | Высокий | Не публиковать / скрыть до проверки модератором |
| 71–100 | Критический | Немедленно ограничить доступ, срочный кейс, эскалация юристу |

⚠️ Пороговые значения — стартовая гипотеза. Корректируются по данным ложных срабатываний после запуска.

#### Что запрещено — ключевые категории

В правилах платформы используется нейтральная юридически корректная формулировка:

> Запрещается размещение информации, распространение которой запрещено или ограничено законодательством РФ, включая экстремистские материалы, призывы к насилию, информацию террористического характера, незаконную рекламу, порнографические материалы, информацию о способах совершения противоправных действий, а также иную информацию, ограниченную для распространения среди несовершеннолетних или неопределённого круга лиц.

**Не удаляется:**
- Субъективные мнения и оценки («мне не понравилась работа», «долго отвечал»)
- Критика без ложных фактов
- Негативные отзывы, не нарушающие правила

---

#### Санкции к аккаунтам

| Санкция | Когда применяется |
|---|---|
| Предупреждение | Первое лёгкое нарушение |
| Скрытие контента | Контент нарушает правила или спорный до проверки |
| Ограничение функции | Запрет комментариев, загрузки аудио, рекламы |
| Временная блокировка | Повторные или существенные нарушения после ручной проверки |
| Постоянная блокировка | Подтверждённые грубые / повторные нарушения после проверки |

> Автоматическая постоянная блокировка допускается только для очевидных технических кейсов: вредоносный спам, боты, фишинг. В остальных случаях — ручная проверка.

---

#### Авторские права

**Принцип:** платформа не устанавливает правообладателя как суд. Платформа оценивает достаточность документов для размещения материала и вправе временно ограничить доступ к спорному материалу.

**При загрузке аудио и портфолио:**
- Пользователь выбирает тип: авторский трек, кавер, ремикс, бит, минусовка, демо, портфолио
- Подтверждает наличие прав или разрешения
- Для кавера / ремикса указывает оригинального автора

**Процесс авторской претензии:**
1. Правообладатель подаёт претензию с документами
2. Неполные претензии уходят на дозапрос
3. Спорный материал временно скрывается
4. Автор получает уведомление и срок для возражений
5. Платформа принимает решение: восстановить, оставить скрытым, удалить

---

#### Реклама и платное продвижение

Всё платное продвижение проходит обязательную **премодерацию** до публикации.

**Запрещено в рекламных материалах:**
- Гарантии попадания в чарты без подтверждения
- Накрутка прослушиваний, лайков, отзывов
- Фишинг, обход сделок платформы
- Реклама запрещённых товаров / услуг
- Использование чужих брендов без права
- Скрытая реклама без маркировки

---

#### Личные сообщения

- Антиспам: лимиты новых диалогов, частоты сообщений, одинаковых текстов
- Шифрование сообщений — не применяется в MVP (бэклог). Уведомление пользователей о просмотре модератором не предусмотрено. Порядок хранения удалённых сообщений прописывается в Политике конфиденциальности
- Проверка ссылок на фишинг и вредоносные домены
- Кнопка «Пожаловаться» на сообщение или диалог
- Блокировка отправителя пользователем
- Антифрод: сигналы — просьбы оплатить напрямую, перейти в мессенджер, отправить код
- Доступ модератора: только пожалованный фрагмент + ограниченный контекст, не вся история

#### Апелляция

- Пользователь получает уведомление о решении с кнопкой «Оспорить»
- Указывает причину, прикладывает документы
- Апелляция не попадает тому же модератору, который принял исходное решение
- Старший модератор или юрист принимает решение
- Результат фиксируется в Audit Log и отправляется пользователю

### Сценарий оформления сделки

При нажатии «Оформить сделку» со страницы услуги или из чата открывается форма. Поля автозаполняются данными из карточки услуги с возможностью редактирования.

**Шаги заполнения формы:**
1. Название сделки (автозаполнено, можно изменить)
2. Услуга — раздел каталога (автозаполнено)
3. Фильтры-атрибуты (автозаполнены из услуги)
4. Сумма заказа — числовой ввод
6. Срок сдачи работы (в днях)
7. Срок приёмки работы (1–5 дней; если не заполнен — система установит 3 дня)
8. Результат — свободный текст, обязательно
9. Количество правок — числовой ввод, обязательно

Внизу формы кнопки «Оформить» и «Отменить».

**«Отменить»:** форма закрывается, пользователь возвращается на страницу услуги.

**«Оформить»:**
- Второму участнику приходит уведомление: «[ФИ] создал сделку по услуге [услуга]. Посмотреть условия и принять в работу вы можете в разделе Мои сделки»
- Сделка появляется у обоих в «Мои сделки» → «Действующие» со статусом «На согласовании»
- Данные из карточки услуги фиксируются в сделке как snapshot

**Статус «На согласовании»:**

У оформившего (User1) — кнопка «Отменить». У второго участника (User2) — «Принять» и «Отклонить».

- User1 нажимает «Отменить» → статус «Отменена», сделка → Архив у обоих. Оба получают уведомление об отмене
- User2 нажимает «Отклонить» → статус «Отменена», сделка → Архив у обоих. Оба получают уведомление об отмене
- User2 нажимает «Принять» → статус «Ожидает оплаты». User1 получает уведомление: «[ФИ] подтвердил сделку [название]. Ожидает оплату»

**Статус «Ожидает оплаты»:**

У исполнителя — «Отменить». У заказчика — «Оплатить» и «Отменить».

- Кто-либо нажимает «Отменить» → статус «Отменена», сделка → Архив у обоих. Оба получают уведомление
- Заказчик нажимает «Оплатить» → открывается окно оплаты через платёжный сервис (безопасная сделка). Деньги холдируются на виртуальном счёте. После подтверждения оплаты:
  - Статус → «В работе»
  - Создаётся деловой чат в разделе «Деловые», название = название сделки
  - В чат приходит системное сообщение: «Сделка зафиксирована! Работа над задачей [название сделки] началась. Удачи!»
  - У исполнителя появляется кнопка «Сдать работу»
  - У заказчика появляются кнопки «Принять» и «На доработку» (неактивны до сдачи работы)

### Сценарий завершения сделки

**Исполнитель нажимает «Сдать работу»** (из статуса «В работе» или «На доработке»):
- Статус → «На проверке»
- Заказчику приходит уведомление: «[ФИ исполнителя] завершил работу по услуге [Название]. Примите работу в течение [срок приёмки] дней»
- Кнопки «Принять» и «На доработку» у заказчика становятся активны
- Система фиксирует `acceptance_deadline` = дата сдачи + срок приёмки

**Заказчик нажимает «Принять»:**
- В деловой чат приходит системное сообщение: «Поздравляем с успешным завершением сделки!»
- Статус → «Завершена», сделка → Архив у обоих
- Счётчик сделок у обоих +1
- Между пользователями автоматически устанавливается профессиональная связь
- Платёжный сервис получает сигнал: деньги размораживаются и распределяются — сумма на руки на реквизиты исполнителя, комиссия на счёт Moooza
- На странице сделки появляется кнопка «Оценить работу» у обоих

**Исполнитель может отменить сдачу работы:**
Если заказчик ещё не нажал «Принять» — исполнитель может нажать «Отменить сдачу». Статус сделки возвращается в «В работе».

**Заказчик нажимает «На доработку»:**
- Открывается окно с полем «Укажите, что именно необходимо доработать» и кнопкой «Отправить»
- В деловой чат приходит системное сообщение: «Работа по услуге [название] требует доработки. Требования: [текст]»
- Статус → «На доработке»
- `revisions_left` −1
- Если `revisions_left` = 0 после финальной сдачи — заказчику приходит уведомление: «[ФИ] предоставил финальный результат по услуге [Название]. Примите работу в течение [срок] или согласуйте с исполнителем изменение количества правок»

**Автозавершение по таймауту:**
Если `acceptance_deadline` наступил, а заказчик не совершил никаких действий — система автоматически переводит сделку в статус «Завершена». Деньги выплачиваются исполнителю, Moooza получает комиссию. Оба получают уведомление.

**Отмена сделки на любом этапе:**
Любой из участников может нажать «Отменить». Оба получают уведомление об отмене.

Вне зависимости от того, кто инициировал отмену и на каком этапе — заказчику возвращается полная сумма за вычетом комиссии платёжного оператора. Исполнитель получает деньги только при успешном завершении сделки.

Если исполнитель понёс затраты до отмены — вопрос компенсации решается между участниками напрямую: через изменение условий текущей сделки, оформление новой сделки или расчёт за пределами платформы. У исполнителя есть возможность оставить отзыв на заказчика и повлиять на его репутацию.

Если сделка была в статусе «В работе» / «На проверке» / «На доработке» в момент отмены — счётчик сделок у обоих +1.

### Редактирование сделки

**До принятия сделки:** редактирование доступно только пользователю, который сформировал сделку. Все поля активны. Кнопка «Сохранить».

**После принятия:** редактирование доступно только по пунктам: Срок сдачи работы, Срок приёмки работы, Количество правок. Остальные поля недоступны. Кнопка «Согласовать изменения» — второй участник получает уведомление: «[ФИ] направил запрос на изменение [пункт] по услуге [название]. Посмотреть и принять вы можете на странице Мои сделки».

На странице «Мои сделки» вверху раздел запросов: название сделки, теги изменяемых пунктов. При нажатии открывается окно «Изменение условий сделки» с прежними и новыми условиями и кнопками «Отклонить» / «Принять».

- «Отклонить» → инициатор получает уведомление об отклонении
- «Принять» → оба получают уведомление об изменении. Данные на странице сделки обновляются. Согласованное число правок прибавляется к остатку (`revisions_left += N`) — пользователь указывает сколько правок добавить, а не новое общее число

### Сценарий оценки пользователя

**Два независимых канала отзыва:**

1. **По сделке** — кнопка «Оценить работу». Доступна при статусах «Завершена» и «Отменена», на любом этапе отмены. Двусторонняя связь **не требуется**.
2. **Внутри связи** — кнопка «Оставить отзыв» на странице связи. Доступна только при установленной **вручную** двусторонней связи.

**Исключение дублей:**
- Если связь установлена вручную → кнопка «Оставить отзыв» в связи есть
- Если связь создана автоматически после сделки → кнопки «Оставить отзыв» в связи **нет** (отзыв оставляется только через сделку)

**Форма отзыва:**
Открывается окно «Поделитесь впечатлениями от взаимодействия». Ниже:
- 10-звёздочная шкала оценки
- Подсказка под звёздами (аналогично Кинопоиску): 10 — Восхитительно, 9 — Отлично, 8 — Очень хорошо, 7 — Хорошо, 6 — Приемлемо, 5 — Посредственно, 4 — Ниже среднего, 3 — Плохо, 2 — Очень плохо, 1 — Ужасно
- Возможность изменить оценку до отправки
- Текстовое поле для отзыва — многострочный свободный ввод, без ограничения по символам
- Кнопка «Отправить»

**После отправки:**
- Отзыв публикуется сразу, без предварительной модерации
- Оценка учитывается в среднем значении оценки пользователя
- Пользователю, которого оценили, приходит уведомление: «Вы получили оценку по сделке [название]» или «Вы получили оценку по взаимодействию в рамках связи с [ФИ] по [предмет]»
- Отзыв попадает в раздел «Отзывы» в поп-апе появляются кнопка «Ответить» (для того, на кого оставлен отзыв), кнопка «Пожаловаться» (для того, на кого оставлен отзыв) и кнопка «Редактировать» (для автора отзыва)

### Редактирование отзыва

Автор отзыва может в любой момент изменить текст отзыва и/или оценку.

**Сценарий:**
1. Автор нажимает кнопку «Редактировать» в поп-апе своего отзыва
2. Открывается форма редактирования с текущими значениями оценки и текста
3. Автор вносит изменения
4. Нажимает «Сохранить» — изменения применяются, отзыв обновляется

Платформа не редактирует отзывы самостоятельно. Содержание отзывов со стороны администрации не изменяется — отзыв либо остаётся в исходном виде, либо удаляется полностью по результатам проверки жалобы.

### Сценарий жалобы на отзыв

Пользователь, на которого оставлен отзыв, может оставить ответ на отзыв и/или пожаловаться на него. Оба действия доступны независимо друг от друга.

**Жалоба на публикацию в Потоке (при нажатии «Пожаловаться» под постом):**

Категории (single select):
- Спам
- Мошенничество / обман
- Оскорбления и угрозы
- Контент для взрослых / неприемлемый контент
- Нарушение авторских прав
- Дезинформация
- Другое

Последствия: только удаление контента (предупреждений нет).

Уведомления:
- Подавший жалобу — получает уведомление с финальным решением
- Автор поста — получает уведомление только при удалении: «Ваша публикация удалена за нарушение правил платформы». При отклонении жалобы — не уведомляется

---

**Жалоба на отзыв:**
1. Пользователь нажимает кнопку «Пожаловаться» в поп-апе отзыва
2. Открывается текстовое поле с подсказкой «Опишите суть претензии и с чем вы не согласны»
3. Пользователь заполняет текст жалобы
4. Нажимает «Отправить» — жалоба отправляется в саппорт

**Обработка жалобы:**
- Отзыв по умолчанию остаётся опубликованным до завершения проверки
- Жалобу рассматривает модератор вручную
- По мере роста платформы и увеличения числа жалоб процесс будет автоматизироваться

**Критерии проверки:**

Модератор проверяет отзыв по следующим признакам:

1. **Нарушение правил платформы** — наличие оскорблений, мата, угроз, персональных данных третьих лиц, спама, рекламы
2. **Клевета и ложные факты** — утверждения о конкретных фактах, которые не соответствуют действительности и могут нанести вред репутации. Субъективные оценки и личный опыт пользователя — не основание для удаления:
   - «Мне не понравилась работа» — оценочное мнение, остаётся
   - «Долго отвечал» — субъективный опыт, остаётся
   - «Человек украл деньги» — потенциальная клевета, требует проверки
3. **Соответствие контексту** — отзыв должен касаться реального взаимодействия в рамках сделки или связи. Отзывы не по теме взаимодействия могут быть удалены
4. **Признаки накрутки или атаки:**
   - Массовые негативные отзывы за короткий период
   - Одинаковые или похожие тексты от разных пользователей
   - Подозрительная активность аккаунта (новый аккаунт, нет истории)
   - Признаки того, что отзыв оставлен конкурентом

**Результат проверки:**
- Нарушений нет → отзыв остаётся без изменений
- Нарушения подтверждены → отзыв удаляется полностью. Платформа не редактирует содержание отзывов
- Автору отзыва и пользователю, подавшему жалобу, приходит уведомление с финальным решением: оставлен или удалён, и обоснование — по какой причине принято это решение

### Сценарий добавления артиста

1. Пользователь нажимает кнопку «Добавить»
2. Открывается окно с полями: Название, Роль, текст «Требуется подтверждение администратором Артиста», кнопка «Запросить подтверждение»
3. **Поле Название:** поиск с ограничением по списку артистов. ⚠️ Тип поля (autocomplete или список с чекбоксами) — требует уточнения. Если артист не найден — результат «Такого артиста не найдено» и кнопка «Создать профиль Артиста»
4. **Поле Роль:** поиск с ограничением по каталогу профессий. ⚠️ Тип поля и логика мультиселекта — требуют уточнения. Пример: ввод «гитара» → выдача «Гитара», «Бас-гитара», «Акустическая гитара». Если ничего не найдено — «Такой роли не найдено». Можно выбрать несколько ролей — подсказка на экране: «Если у вас несколько ролей, укажите каждую»
5. Кнопка «Запросить подтверждение» неактивна, пока не заполнены оба поля. После заполнения становится активной
6. При нажатии запускается сценарий подтверждения добавления участника артиста

### Сценарий подтверждения добавления участника артиста

После нажатия «Запросить подтверждение» пользователю всплывает окно: «Запрос направлен! После подтверждения Артист будет добавлен в ваш список Артистов».

Администратор получает уведомление: «[ФИ] запрашивает подтверждение роли [Роль] в [Название артиста]». При нажатии открывается окно подтверждения с кнопками «Отклонить» и «Подтвердить».

- **«Подтвердить»:** пользователю приходит уведомление. В его профиле появляется плитка артиста. В профиле артиста добавляется новый участник
- **«Отклонить»:** пользователю приходит уведомление «Ваш запрос на подтверждение роли [Роль] в [Название артиста] отклонён»

### Сценарий установления связи

1. User1 нажимает «Установить связь» в профиле User2 → иконка меняет цвет, кнопка становится неактивной
2. User1 попадает в отдельное окно, где:
   - Выбирает роль — **single select**: исполнитель / заказчик / коллега
   - Выбирает предмет связи — **поисковая строка с ограничением по каталогу услуг**. Найденный предмет добавляется тегом. Можно добавить несколько предметов — количество не ограничено (поиск → добавить тег → поиск → добавить тег и т.д.)
   - Если ролей несколько — можно добавить ещё одну роль и указать к ней свой предмет связи
3. Отправляет запрос
4. Связь отображается в списке связей User1 (не подтверждена, иконка серая). Счётчик связей User1 +1
5. User2 получает уведомление. В разделе «Запросы» → «Связи» у User2 появляется запрос
6. У User1 в разделе «Запросы» → «Связи» появляется запрос с кнопкой «Отменить»

**Отмена User1:** связь пропадает из списков и запросов обоих, кнопка снова активна, счётчик −1.

**User2 нажимает «Отклонить»:** связь пропадает из списков и запросов обоих, User1 получает уведомление об отклонении, кнопка снова активна, счётчик −1.

**User2 нажимает «Принять»:** у User2 связь появляется с фиолетовой иконкой, запрос пропадает у обоих, счётчик связей User2 +1. У User1 иконка из серой становится фиолетовой. User1 получает уведомление о подтверждении.

После установления связи у обоих появляется кнопка «Оценить взаимодействие» на странице связи.

### Сценарий добавления в друзья

1. User1 нажимает «Добавить в друзья» в профиле User2 → иконка меняет цвет, кнопка неактивна у обоих
2. User2 получает уведомление. В «Запросах» у User2 появляется запрос с кнопками «Принять» и «Отклонить»
3. У User1 в «Отправлено» появляется запрос с кнопкой «Отменить»

**User1 нажимает «Отменить»:** запрос пропадает у обоих, кнопка снова активна.

**User2 нажимает «Отклонить»:** запрос пропадает у обоих, User1 получает уведомление, кнопка снова активна.

**User2 нажимает «Принять»:** User2 появляется в списке друзей User1 и наоборот. Запрос пропадает. User1 получает уведомление. Счётчик друзей у обоих +1. Кнопка «Добавить в друзья» меняется на «Удалить из друзей».

### Сценарий удаления из друзей

1. User1 нажимает «Удалить из друзей» в профиле User2
2. Всплывает окно: «Вы точно хотите удалить [ФИ User2]?» с кнопками «Отменить» и «Удалить»
3. «Отменить» → окно закрывается, ничего не происходит
4. «Удалить» → оба пропадают из списков друзей друг друга. Счётчик друзей у обоих −1

### Сценарий добавления в избранное

1. User1 нажимает «Добавить в избранное» в профиле User2
2. Иконка меняет цвет на фиолетовую
3. User2 получает уведомление, что его добавили в Избранное
4. В разделе «Избранное» у User1 появляется User2

### Автоматическое установление связи

Происходит при завершении сделки.

**Если связи между пользователями ранее не было:** счётчик связей у обоих +1, пользователи появляются друг у друга в разделе «Связи» сразу с фиолетовой иконкой.

**Если связь уже была:**
- Новая роль → новый блок роли и деятельности в карточке связи
- Новая деятельность в рамках существующей роли → добавляется тег деятельности
- Деятельность уже была → счётчик взаимодействий по ней +1
- Если был pending-запрос на связь — он закрывается автоматически при завершении сделки

**Роли назначаются автоматически:**
- В рамках сделки: «Исполнитель» и «Заказчик»
- Между участниками Артиста: «Коллеги»

**Вид деятельности/услуга:**
- В рамках сделки — из содержания сделки
- Между участниками Артиста — роли участников

### Режим редактирования профиля

**Блок Основная информация:**
- Аватарка — кнопка смены аватарки, открывается системный компонент выбора фото/файлов
- Имя — поле с рамкой, редактируется при нажатии
- Фамилия — свободный ввод текста, редактируется при нажатии
- Никнейм — свободный ввод текста; проверяется на уникальность: если занят — выделяется красным, надпись «Никнейм занят, введите другой»
- Страна — поиск с ограничением по каталогу, выпадающий список (autocomplete)
- Город — поиск с ограничением по каталогу, выпадающий список (autocomplete). Подсказка: «Если вы не нашли свой город, выберите ближайший к вам»
- Дата рождения — свободный ввод цифр с маской в формате 00.00.0000, точки проставлены заранее

**Блок О себе:**
- Многострочное текстовое поле, свободный ввод, ограничение 100 символов
- Счётчик символов: «0/100» до заполнения, «88/100» в процессе

**Блок Профессии:**
- Поиск с ограничением по каталогу профессий (autocomplete с выпадающим списком). Подсказка: «Начните вводить название и выберите из списка. Добавить можно до 10 профессий»
- При выборе профессии раскрываются её фильтры — мультиселект (можно выбрать несколько значений)
- Кнопка «Сохранить» в нижней части окна фильтров — профессия появляется в списке
- У каждой профессии в режиме редактирования: название слева, кнопки «Редактировать» и «Удалить» справа
- При «Удалить» — окно подтверждения «Точно хотите удалить?»
- Если профессия не найдена — «Такой профессии не найдено, напишите в поддержку»

**Блок Соцсети и контакты:**
- Список соцсетей — иконки и строки для ввода ссылки/ID/названия канала. Свободный ввод текста
- Контакты — иконки мессенджеров, электронной почты, телефона и строки ввода. Свободный ввод текста

**Прогресс заполнения профиля:** в профиле отображается процент заполнения с подсказкой что осталось добавить.

**Кнопка «Превью»:** в своём профиле — позволяет посмотреть профиль глазами других пользователей, с учётом настроек видимости.

**Автосохранение:** при переходе из блока в блок данные сохраняются автоматически.

**Кнопка «Завершить редактирование»:** выход из режима редактирования, все изменения отображаются в профиле. Счётчик профессий обновляется.

При случайном закрытии страницы — данные блоков, из которых был совершён переход (автосохранение), сохраняются.

### Реферальная программа

Добавляется в первом обновлении после MVP.

**Концепция:** «Давайте делать индустрию вместе. Приглашай друзей и коллег. Получай бонусы.»

**Механика:**
- У каждого пользователя — персональная реферальная ссылка + кнопка «Скопировать»
- Приглашённый регистрируется по ссылке → у пригласившего засчитывается +1

**Прогресс-бары:**
- Бар 1 — прогресс к бонусу: 0 из 10. При достижении 10 обнуляется
- Бар 2 — прогресс к статусу Амбассадора: 0 из 100

**Бонусы:**
- За каждые 10 приглашённых — 1 месяц подписки Pro (включается автоматически)
- За 100 приглашённых — статус Амбассадора

**Раздел «Полученные бонусы»:** список полученных бонусов.

**Кнопка в шапке:** открывает окно с условиями реферальной программы. В MVP кнопки нет — добавляется вместе с реферальной программой.

⚠️ Предварительное описание — условия могут измениться до реализации.

### Сценарий жалобы на публикацию

При нажатии «Пожаловаться» под постом в Потоке:
1. Открывается окно с выбором категории жалобы (single select — те же категории, что и в жалобе на пользователя)
2. Текстовое поле для описания причины — многострочный ввод, без ограничения по символам
3. Кнопка «Отправить» — жалоба уходит в саппорт и в административную панель
4. Подтверждение: «Ваша жалоба принята и будет рассмотрена в течение 3 дней»

Обработка — по тем же правилам что и жалоба на пользователя (см. раздел «Модерация»).

🔗 Связано с: «Модерация», «Сценарий жалобы на пользователя».

Платный раздел — один из инструментов монетизации платформы. Доступ предоставляется по подписке или разово.

**Общая идея:**
Пользователь получает детальную аналитику по своей деятельности на платформе и данные по рынку в разрезе своих профессий и услуг.

**Планируемые блоки (требуют детальной проработки):**
- Заработок на платформе — сколько заработано за период, динамика
- Активность — количество сделок, средний чек, скорость получения заказов
- Когортный анализ
- Сравнение с конкурентами — по каждой услуге: средняя стоимость (мин/макс/средняя) по рынку, среднее количество заказов у других исполнителей этой услуги, скорость получения заказов, где пользователь находится относительно других
- Данные по рынку в целом

### Кейсы на странице услуги

Блок «Кейсы» появляется на странице услуги и в форме создания/редактирования услуги. Позволяет исполнителю подтвердить услугу конкретными работами.

**Блок «Кейсы» при создании услуги:**

Текст блока: «Добавьте работы, которые подтверждают эту услугу»

Два варианта добавления:

**Вариант 1 — Выбрать из портфолио:**
Открывается список работ из портфолио пользователя. Он отмечает галочками подходящие работы — каждая работа может быть привязана к одной или нескольким услугам. Отмеченные работы появляются в блоке «Кейсы» на странице услуги.

**Вариант 2 — Добавить новый кейс:**
Пользователь заполняет форму:
- Название кейса
- Описание
- Роль в проекте
- Что было сделано
- Ссылка на результат
- Фото / изображение
- Жанр
- Год
- Заказчик / артист / проект
- Можно ли показывать публично (да / нет)

После сохранения новый кейс:
- Появляется в блоке «Кейсы» на странице услуги
- Автоматически попадает в общее портфолио пользователя

⚠️ Детали отображения блока «Кейсы» на странице услуги и в портфолио требуют отдельной проработки.

### Уведомления — поведение при клике

- При клике на уведомление пользователь переходит на соответствующий экран (страница сделки, чат, раздел «Запросы» и т.д.)
- Уведомление помечается как прочитанное при клике
- Непрочитанное — жирный шрифт, прочитанное — обычный
- Кнопка «Прочитать всё» — помечает все уведомления прочитанными

**Push-уведомления:**
- Включены по умолчанию, отдельного запроса разрешения нет
- В настройках — список всех типов уведомлений с бегунком вкл/выкл
- По умолчанию все типы включены
