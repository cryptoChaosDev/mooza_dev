# Отчёт по аудиту безопасности Moooza

**Дата:** 2026-06-12 · **Ветка:** dev
**Метод:** многоагентный аудит — 8 измерений безопасности (фан-аут) → состязательная верификация каждой находки независимым скептиком → синтез. 48 агентов, 39 кандидатов → **19 подтверждено**, 20 отклонено как ложные срабатывания.
**Объём:** серверная часть (Express+TypeScript+Prisma), клиент (React SPA), инфраструктура, зависимости.

- **CI `npm audit`** и **аудит-лог админ-действий** — организационные улучшения.

## 1. Executive Summary

Общий уровень защищённости Moooza — **средний, с заметным базовым фундаментом** (JWT-аутентификация, helmet+CSP, DOMPurify для пользовательского контента, rate-limiting, нормализованный поиск). **Критических уязвимостей с немедленной угрозой полного захвата системы не обнаружено.** Подтверждено **19 проблем**: 0 critical, **10 high**, **9 medium**. Главный системный недостаток — **отсутствие выходного экранирования (output encoding)** при вставке пользовательских данных в неэскейпленные контексты (HTML OG-эндпоинт, Telegram-уведомления) и **повсеместный паттерн «check-then-act» без атомарности** в бизнес-логике (сделки, никнеймы, рефералы, отзывы).

**Главный вывод:** продукт безопасен в типовых сценариях, но до выкатки в PROD требует устранения XSS в OG-эндпоинте, закрытия прямого доступа к загруженным файлам, обновления уязвимых npm-зависимостей, усиления brute-force-защиты кодов и перевода критичных бизнес-операций на транзакции/атомарные WHERE.

## 3. Детальный разбор

### 🔴 HIGH

#### №1. Stored XSS / HTML-инъекция в OG-эндпоинте профиля
**Где:** `server/src/index.ts:169-197` (вставка), `server/src/routes/users.ts:435-478` (нет санитизации на входе).
**Суть.** `GET /api/og/profile/:userId` строит HTML через template-литералы, напрямую подставляя `firstName`, `lastName`, `bio`, `city`, `avatar`, `url` без HTML-entity-кодирования. Валидация на входе проверяет только тип/длину. Данные сохраняются в БД и попадают в неэскейпленный HTML (атрибуты `content="..."`, тело `<a href="...">name</a>`).
**Сценарий.** `firstName` = `Test"><script>alert(1)</script>` или `<img src=x onerror=alert(1)>`; либо `bio` со спуфингом OG-тегов под фишинг. CSP `unsafe-inline` (№4) позволяет исполнение инлайн-скрипта.
**Оговорка верификатора.** Краулеры соцсетей JS не исполняют, а эндпоинт делает мгновенный `meta refresh`-редирект — это снижает риск исполнения JS в браузере конечного пользователя, но **не устраняет** XSS (прямой просмотр URL / превью-сервисы) и OG-спуфинг. Итог — **High**.
**Фикс.** HTML-entity-кодирование ВСЕХ полей (`he`/`escape-html`/`html-entities`) перед вставкой; дополнительно санитизировать имя/bio при записи.

#### №2. Неавторизованный прямой доступ к загруженным файлам (IDOR)
**Где:** `server/src/index.ts:132`, `server/src/routes/users.ts:170`.
**Суть.** Все файлы отдаются статикой `/uploads/*` без аутентификации. Публичный профиль (`optionalAuthenticate`) отдаёт прямые URL портфолио. У `PortfolioFile` нет поля видимости.
**Сценарий.** Собрать userId → дёрнуть публичный профиль без авторизации → извлечь URL портфолио → скачать любые файлы напрямую; возможен энумерейшн.
**Фикс.** Авторизованный маршрут `GET /api/files/:fileId` с проверкой владения/прав; для публичных аватаров — подписанные URL или явный публичный флаг.

#### №3. Уязвимые npm-зависимости сервера (12, из них 5 high)
**Где:** `server/package.json`.
**Суть.** `npm audit`: `esbuild@0.27.2` (HIGH), `path-to-regexp@0.1.12` ReDoS (HIGH), `socket.io-parser@4.2.5` (HIGH), `express-rate-limit@8.2.1` — обход через IPv4-mapped IPv6 (HIGH), `ws@8.18.3` (MOD), `qs`, `ip-address` (MOD).
**Сценарий.** ReDoS через крафтовый URL; обход rate-limit на dual-stack сетях (усиливает №5/№11/№12); DoS через socket.io.
**Фикс.** `npm audit fix`; минимум: `express-rate-limit>=8.6.0`, `path-to-regexp>=0.1.13`, `socket.io-parser>=4.2.6`, `ws>=8.20.1`, `esbuild>=0.28.1`. `npm audit` в CI.

#### №4. CSP разрешает `'unsafe-inline'` для скриптов и стилей
**Где:** `server/src/index.ts:80-81`.
**Суть.** `scriptSrc`/`styleSrc` с `'unsafe-inline'` сводят на нет защиту CSP против XSS; в связке с №1 позволяют исполнение инлайн-`<script>`.
**Фикс.** Убрать `'unsafe-inline'` из `scriptSrc` (nonce/hash). Для `styleSrc` — ниже приоритет (React использует инлайн-стили), потребует рефакторинга.

#### №5. Brute-force кода верификации (лимит по IP, не по email)
**Где:** `server/src/middleware/rateLimiter.ts:88-97`; `auth.ts:223/267/285/1064/1073`.
**Суть.** `codeLimiter` — 10 попыток/15 мин **на IP**, не на email. Код 6 цифр (900k), хранится plaintext. Распределённая атака N IP умножает попытки против одного email; цикл `/register` пере-генерирует код, сбрасывая контекст.
**Фикс.** Лимит по email+IP; код 8+ цифр/алфавитно-цифровой; хранить хэш; ограничить пере-генерации; TTL 5-10 мин.

#### №6. DoS через N+1 в `GET /api/messages/unread/count`
**Где:** `server/src/routes/messages.ts:68-95`.
**Суть.** Все членства без лимита + **последовательный** `count()` по каждой беседе. 1000 бесед = 1001 запрос → таймаут, исчерпание пула.
**Фикс.** Один агрегированный запрос (`GROUP BY conversationId`) / Redis-кэш / пагинация с лимитом.

#### №7. Race condition в переходах статуса сделки
**Где:** `server/src/routes/deals.ts:120-134` (и `/reject`, `/cancel`, `/pay`, `/submit`, `/approve`, `/revision`).
**Суть.** «check-then-act» без атомарности: `findUnique` → проверка `status` в коде → отдельный `update`. Конкурентные `accept`+`cancel` оба проходят проверку.
**Фикс.** Атомарный WHERE по статусу:
```ts
const r = await prisma.deal.updateMany({ where: { id, status: 'PENDING' }, data: { status: 'AWAITING_PAYMENT' } });
if (r.count === 0) return res.status(409).json({ error: 'Статус изменился' });
```
Применить ко всем переходам.

#### №8. Создание отзыва без проверки сделки/связи (fake reviews)
**Где:** `server/src/routes/reviews.ts:35-67`; `prisma/schema.prisma:639-662`.
**Суть.** `POST /api/reviews` проверяет лишь `targetId != userId`, поля и диапазон. Нет проверки ACCEPTED `Connection` / завершённого `Deal`; `dealId` nullable, FK нет.
**Сценарий.** Несколько аккаунтов заваливают конкурента отзывами без взаимодействия (репутационная атака).
**Фикс.** Серверная валидация связи (ACCEPTED Connection для `type=connection`; COMPLETED/REVIEW Deal для `type=deal`); FK-ограничения; rate-limit на отзывы.

### 🟠 MEDIUM

#### №9. HTML-инъекция в Telegram-уведомлениях
**Где:** `server/src/utils/telegram.ts:28-66`.
**Суть.** Пользовательские данные интерполируются в сообщения с `parse_mode:'HTML'` без экранирования. `firstName` = `</b><a href="http://attacker">Click</a><b>` внедряет кликабельную ссылку в админ-уведомление (фишинг администратора).
**Фикс.** `escapeTgHtml()` для всех данных в `tgEvent.*`.

#### №10. SSRF через получение OG-метаданных
**Где:** `server/src/utils/streamMetadata.ts:53-91`; вход — `releases.ts:65`, `clips.ts:64` (под `authenticate`).
**Суть.** `tryOpenGraph()` делает `fetch()` по произвольному URL без валидации. Аутентифицированный пользователь через `platform` без oEmbed (VK/YANDEX/APPLE/VK_VIDEO) → default-ветка → `tryOpenGraph`.
**Сценарий.** `{platform:'VK', url:'http://localhost:5432'}` / `169.254.169.254` → доступ к внутренним сервисам. Требует JWT → Medium.
**Фикс.** Блокировать приватные/loopback/link-local; резолвить hostname и проверять IP (в т.ч. после редиректов — DNS-rebinding); лимит редиректов; предпочтительно allowlist платформ.

#### №11. Перечисление аккаунтов через `/check-email`
**Где:** `server/src/routes/auth.ts:143-153`.
**Суть.** Возвращает `{available}`, раскрывая существование аккаунта; только глобальный `apiLimiter` (5000/15 мин) → база 100k за ~5 часов.
**Фикс.** Строгий лимит (5-10/час/IP) или аутентификация; балансировать с UX-проверкой при регистрации.

#### №12. Перечисление через коды ответа `/reset-password`
**Где:** `server/src/routes/auth.ts:1070-1074`.
**Суть.** 404 для несуществующего vs 400 для неверного кода — различимы. (`/forgot-password` сделан корректно — 200 для всех.)
**Фикс.** Единообразный ответ для обоих случаев.

#### №13. DoS через N+1 в `GET /api/messages/conversations`
**Где:** `server/src/routes/messages.ts:120-159`.
**Суть.** `Promise.all` по всем членствам с `count()` на беседу. 500 бесед = 500 параллельных запросов; смягчён `apiLimiter`.
**Фикс.** Агрегированный запрос / Redis / пагинация. Чинить вместе с №6.

#### №14. Низкая энтропия 6-значного кода
**Где:** `server/src/routes/auth.ts:222-223`.
**Суть.** 900k вариантов (~19.78 бит) ниже стандарта 8+ цифр. В связке с №5 усиливает brute-force.
**Фикс.** 8-значный/алфавитно-цифровой код; email-лимит; TTL 5-10 мин. Связано с №5.

#### №15. TOCTOU уникальности никнейма
**Где:** `server/src/routes/users.ts:460-466` (и `admin.ts:679-686`).
**Суть.** `findFirst`+`update` не атомарны. UNIQUE-индекс на `nicknameNorm` спасает данные, но второй конкурент падает с raw-ошибкой БД вместо 409.
**Фикс.** Атомарный апдейт с условием `NOT { nicknameNorm }` и/или обработка `P2002` как 409.

#### №16. Не-атомарное «сжигание» реферальной ссылки
**Где:** `server/src/routes/auth.ts:378-390`.
**Суть.** Ссылка «сжигается» отдельной операцией после `user.create`, в `.catch()` без ретрая. При сбое burn+rollback пользователь создан, а ссылка остаётся «свободной».
**Фикс.** Перенести «сжигание» в один `$transaction` с созданием пользователя.

#### №17. Self-referral farming (накрутка Pro)
**Где:** `server/src/routes/auth.ts:329-330`; `isValidReferral` `:67-81`; `pro.ts:121`. (orig low → **medium**, т.к. платная монетизация.)
**Суть.** `isValidReferral()` проверяет лишь существование реферера, без анти-самореферала. Атакующий регистрирует B/C/… по своей ссылке → бесплатные месяцы Pro; `registerLimiter` обходится VPN.
**Фикс.** Запрет самореферала/циклов; rate-limit начислений Pro по IP/устройству; учёт «качества» приглашённых (подтверждённый email + активность).

## 4. Оценка защищённости по классам атак

| Класс атаки | Статус | Комментарий |
|-------------|--------|-------------|
| Brute-force / credential stuffing | 🟡 частично | `authLimiter`/`codeLimiter` есть, но лимит кода по IP не email; код 6 цифр plaintext + цикл пере-генерации (№5, №14). |
| XSS | 🟡 частично | Контент защищён DOMPurify, НО OG-эндпоинт без экранирования + CSP `unsafe-inline` (№1, №4). |
| SQL-инъекции | 🟢 хорошо | Prisma (параметризация), нормализованный поиск. Подтверждённых нет. |
| IDOR / контроль доступа | 🔴 слабо | `/uploads/*` без авторизации, портфолио-файлы перечислимы (№2). |
| SSRF | 🟡 частично | Произвольный `fetch` в OG-метаданных, только для аутентифицированных; нет валидации приватных IP (№10). |
| Загрузка файлов | 🟡 частично | 7 multer-конфигов, блок-лист для чата, рендер через `<img>`. Проблема — доступ (№2), не сам аплоад. SVG/MIME-спуфинг признаны неэксплуатируемыми. |
| CSRF | 🟢 хорошо | JWT в заголовке `Authorization`, не в куках — классический CSRF неприменим. |
| DoS | 🟡 частично | N+1 в `messages` (№6, №13) при глобальном лимите; ReDoS-риск `path-to-regexp` (№3). |
| Утечка секретов | 🟢 хорошо | Явных утечек в коде не подтверждено; env-конфиг. (Косвенно — plaintext-коды, №5.) |
| Перечисление аккаунтов | 🟡 частично | `/check-email`, `/reset-password` раскрывают аккаунт (№11, №12). |
| Эскалация привилегий | 🟢 хорошо | Админ-роуты под `authenticate + requireAdmin`; повышения прав не найдено. |
| Бизнес-логика / платежи | 🔴 слабо | Гонки сделок (№7), fake reviews (№8), TOCTOU ника (№15), не-атомарный реферал (№16), self-referral farming (№17). |

## 5. Что сделано хорошо

- **JWT в заголовке `Authorization`, а не в куках** — классические CSRF неприменимы по дизайну.
- **DOMPurify для пользовательского контента** (`PostContent.tsx`) со строгим `ALLOWED_TAGS`; TipTap на безопасный HTML.
- **Prisma ORM** — параметризация, подтверждённых SQL-инъекций нет.
- **helmet + CSP** настроены (`objectSrc:'none'`, `scriptSrc:'self'`), HSTS включён (кандидат «HSTS не настроен» отклонён).
- **Базовый rate-limiting** (`authLimiter`, `registerLimiter`, `codeLimiter`, глобальный `apiLimiter`).
- **`path.basename()` при удалении файлов** блокирует path traversal (кандидат отклонён как FP).
- **Блок-лист опасных расширений** для чат-вложений.
- **Админ-доступ корректно защищён** (`requireAdmin`).
- **Команда умеет работать с транзакциями** (`deals.ts:345`, `artists.ts`) — нужно распространить на остальные операции.

> Отклонённые кандидаты (FP): legal-документы через `dangerouslySetInnerHTML` — static-файлы из репо (defense-in-depth, не runtime-XSS); SVG-аплоад и MIME-спуфинг неэксплуатируемы (рендер `<img>`, CSP); CORS origin-less и `/check-nickname` — приемлемый риск.
